# Project Review — 2026-06-10

Scope: general progress, functionality, UX, code quality, performance, and real-life usage
by humans and agents. Produced from a four-lens review (runtime reliability, code quality,
human UX/docs, agent-driveability) plus firsthand operating experience migrating the
digitech-next PR review from a poller to a webhook router.

## Overall state

The core is healthy: a working trigger substrate (schedules / polls / webhooks → actions),
a router that binds PR lifecycles to long-lived agent sessions, lean dependencies
(2 runtime deps), and real production use — two live PR routers as of today. Tests pass
(58) and the recent commit cadence shows steady feature progress.

The project has crossed from "building features" into the phase where the gaps that matter
are **lifecycle reliability, observability, and the operating contract** — not missing
functionality.

---

## 1. Router binding lifecycle (highest priority)

The router is now load-bearing for daily PR reviews, and its failure modes leak long-lived
agent sessions or silently stop activity flow.

| # | Finding | Severity | Evidence |
|---|---------|----------|----------|
| 1.1 | `idleTtl` is parsed but enforced nowhere. A missed close event (daemon down, relay outage, GitHub delivery failure) leaks the bound bee session forever. | Critical | `src/types.ts:108` defines it; no consumer in `src/router.ts` or `src/daemon.ts`; no GC analogous to `gcTemporaryHooks` |
| 1.2 | No reconciliation: nothing periodically verifies that active bindings still correspond to open PRs (e.g., `gh pr view` sweep). | Critical | absent from `src/daemon.ts` tick |
| 1.3 | Errored bindings are dead ends: if `onOpen` fails once (transient hive error), the binding sticks at `errored` and is never retried; only a re-`opened` event recreates it. | High | `src/router.ts` createBinding error path; already-bound check skips `errored` but nothing re-drives it |
| 1.4 | `routeToBinding` has no try/catch — a failed `hive send` propagates without recording an error on the binding, so activity silently stops flowing. | High | `src/router.ts:163-165` |
| 1.5 | No webhook redelivery dedup (`x-github-delivery` ignored). The already-bound check prevents duplicate spawns, but redelivered *activity* events hit the bee twice. | Medium | `src/webhook.ts` validates signature only; `src/satellite.ts` forwards the header but never uses it |

**Recommended fix:** a daemon-tick binding GC — enforce `idleTtl`, retry or expire errored
bindings, reconcile active bindings against actual PR state — plus a small delivery-GUID
dedup cache.

## 2. Observability — "why didn't it fire?" is unanswerable

Verified firsthand today:

| # | Finding | Severity | Evidence |
|---|---------|----------|----------|
| 2.1 | `~/.pollinate/daemon.log`, `daemon.out.log`, `daemon.err.log` were all empty while the daemon was actively processing events. `pol daemon logs` is correspondingly useless. | High | `src/service.ts:83-98` cats plist-configured files; daemon writes mostly to ledger |
| 2.2 | `pol get <trigger>` hides the router config — shows only `github-pr · open 3 close 2`. The `openWhen` filter, bee, prompts, and actions are invisible without reading the TOML. | High | `src/cli.ts` `describeRouter()` |
| 2.3 | Webhook signature rejections are not ledgered at all (filtered and dropped events are). | Medium | `src/webhook.ts:179-195` |
| 2.4 | `pol edit` doesn't validate the TOML on editor exit; malformed files surface only as a ledger `reload_errored` event the user never sees. | Medium | `src/cli.ts:224-230`, `src/daemon.ts:76-81` |
| 2.5 | No aggregate drop/reject counters; answering "why didn't my trigger fire?" requires grepping `ledger.jsonl` by hand. | Medium | — |

**Recommended fixes:** render full router detail in `pol get`; wire daemon logging
properly; ledger signature rejections; validate TOML after `pol edit`; add drop/reject
counters to `pol status` (or a `pol why <trigger>`).

## 3. Agent-driveability — ironic gap for an agent-native tool

Pollinate orchestrates agents, but an agent operating *pollinate* must read the source.
Today's migration required reading `config.ts` and `cli.ts` to learn the TOML schema and
template variables.

| # | Finding | Severity | Evidence |
|---|---------|----------|----------|
| 3.1 | No CLAUDE.md / AGENTS.md. The TOML trigger schema, template variable reference (`{{pr_number}}`, `{{pr_author}}`, `{{binding.target}}`, …), ledger event types, and the `<!-- pollinate-router -->` loop-prevention contract are only discoverable in code. | High | no agent docs in repo |
| 3.2 | `--json` coverage is spotty across commands (e.g., `hooks`, `daemon logs`, parts of `hook wait`). | Medium | `src/cli.ts` `print()` call sites |
| 3.3 | `add` / `create` silently overwrite existing triggers; no `--if-not-exists` / `--force`, no distinct exit code. Re-run automation can clobber config. | Medium | `src/store.ts:85-95`, `src/cli.ts` cmdCreate |
| 3.4 | `--dry-run` exists only for `trigger` and the github commands, not `create`/`add`. | Medium | `src/cli.ts:234-241`, `:710-760` |
| 3.5 | Spawned bees get rendered markdown only; structured payload (JSON context env var, binding metadata) would let them act without re-fetching. | Low | `src/context.ts:19-23`, `src/router-plugins/github-pr.ts` |
| 3.6 | No HTTP query API (`GET /status`, `/bindings`, `/jobs`) — agents must shell out or read state files. | Low | `src/webhook.ts:72-74` rejects non-POST |

**Recommended fix:** a one-day pass writing the agent contract doc + uniform `--json` +
idempotency flags. Agents are this project's primary users; this pays for itself
immediately.

## 4. Code quality

Three items before everything else:

| # | Finding | Severity | Evidence |
|---|---------|----------|----------|
| 4.1 | `parseHiveHandle` takes the first whitespace-delimited token of the first non-empty stdout line. One warning line from hive and the router binds to garbage, silently. Prefer `hive spawn --json`, or validate the handle shape (`^[A-Za-z0-9._:-]+$`) and fail loudly. | High | `src/actions.ts` `parseHiveHandle` |
| 4.2 | Webhook-derived data (PR titles, comment bodies — authored by anyone on GitHub) is templated into shell command strings for `hive send`. `shellQuote` covers common cases; argv-based `spawn()` eliminates the class. Related: comment bodies are prompt-injection surface for the bee — read-only `allowedTools` on review triggers is the right mitigation, keep it tight. | High | `src/actions.ts` honeybee command assembly |
| 4.3 | Test coverage is inverted: integration-heavy tests with copy-pasted PATH-hijacking hive stubs, while `config.ts`, `templates.ts`, and `actions.ts` — the highest-bug-density modules — have zero unit tests (16 of 26 src files untested). | High | compare `src/*.ts` vs `tests/*.ts`; stub duplicated in `tests/router.test.ts`, `tests/delivery-context-actions.test.ts` |

Secondary:

- `cli.ts` is ~1,700 lines and growing — split by command family (`triggers`, `jobs`,
  `hooks`, `github`, `daemon`) on next major touch.
- `normalizeFilter` and friends cast unvalidated TOML (`raw as Filter`); ad-hoc
  `AnyRecord` validation throughout `config.ts` lets typos pass silently.
- Error handling is inconsistent: `.catch(() => [])` swallows readdir failures
  (`src/store.ts:69`); warnings and errors share the `job.error` field.
- `pol github create-pr-router` defaults to the `codex` bee with no `--bee` flag and has
  no flag for `openWhen` author filtering — the flagship command required hand-written
  TOML for today's (presumably common) use case.

## 5. Reliability hardening (lower urgency)

| # | Finding | Severity | Evidence |
|---|---------|----------|----------|
| 5.1 | Poll cursor advances even on partial/failed fetch in some paths; no error backoff — a persistently failing poll retries at full cadence. | Medium | `src/poll.ts:40-88` |
| 5.2 | Queued-job persistence races daemon crash; jobs ledgered but never re-enqueued can vanish. | Medium | `src/delivery.ts:172-187`, restore path `:57-74` |
| 5.3 | `withFileLock` stale-lock removal has a TOCTOU race (both processes can conclude the lock is stale); lock file PID is never liveness-checked. | Medium | `src/fsx.ts:105-133` |
| 5.4 | `updateJob` is an unlocked read-modify-write shared by CLI and daemon. | Medium | `src/store.ts:187-193` |
| 5.5 | No webhook body size limit → OOM-able endpoint. | Medium | `src/webhook.ts:228-235` |
| 5.6 | `ledger.jsonl` grows unboundedly and is read in full by CLI commands; no rotation. | Medium | `src/store.ts:241-250` |
| 5.7 | Triggers with no secret accept any POST to their path; fine by design, but worth a warning at creation time. Plaintext secrets in TOML are accepted without nudging toward `env:NAME` (files are 0600, which helps). | Low | `src/webhook.ts:183-191`, `src/secrets.ts` |

## 6. Docs / progress vs. plan

- `openWhen` (added today) is implemented and tested but documented only in
  `docs/pr-router.md`'s option list — absent from README and `pol help`.
- README lacks a first-run path (create trigger → dry-run → daemon install → verify) and
  a "common patterns" section (author-filtered reviews, batching, relay setup).
- The router plugin extension story (`pol routers init`, `~/.pollinate/router-plugins/`)
  is solid but undiscoverable from README.

## Priority punch list

1. **Binding GC + reconciliation** — enforce `idleTtl`, retry/expire errored bindings,
   verify bound PRs are still open. Prevents leaked agent sessions, the failure mode
   most likely to actually occur. (§1)
2. **AGENTS.md / CLAUDE.md contract doc** — TOML schema, template variables, ledger
   events, marker convention. (§3.1)
3. **Observability pass** — `pol get` full router detail, daemon logging, signature-reject
   ledgering, post-edit validation. (§2)
4. **Harden the hive seam** — structured spawn output or strict handle validation;
   argv-based exec. (§4.1, §4.2)
5. **Unit tests for `config.ts` / `templates.ts` / `actions.ts`** + a shared hive test
   stub. (§4.3)
